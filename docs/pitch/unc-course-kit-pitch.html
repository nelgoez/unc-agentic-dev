Caso real · 16 de junio 2026

Un error crítico en producción

Curso 269 — "Aprendiendo a caminar en Python - Certificación 1"

🔴 Módulo 3

Bloqueado permanentemente

Not available unless:
The activity "Notebook Funcion-Lambda" is marked complete

Esa actividad NO EXISTE en el curso

🔴 Cierre

Bloqueado en cascada

Requería 6 actividades del Módulo 3... que estaba bloqueado.

Deadlock: ninguna actividad se puede completar

Detección: un estudiante reportó no poder avanzar.
Impacto: debugging reactivo, corrección urgente en producción, experiencia del estudiante dañada.

Este es el caso que desencadenó esta propuesta. El 16 de junio, el curso 269 tenía un error crítico: el Módulo 3 requería una actividad llamada Notebook Funcion-Lambda para desbloquearse. Esa actividad nunca existió. Probablemente alguien la renombró, o era un PDF que no estaba trackeado como actividad. El resultado es que todo el resto del curso quedó bloqueado. Y nadie lo supo hasta que un estudiante lo reportó.

Root cause analysis

¿Por qué pasó?

🏗️ Creación manual

Las dependencias se configuran una por una en Moodle. No hay validación cruzada entre lo que se escribe y lo que existe.

🔍 Sin revisión

No hay un paso de peer review antes de publicar. El prompter crea, configura y publica solo.

📋 Sin checklist

No hay una lista de verificación que asegure que cada actividad existe antes de usarla como condición de acceso.

📊 Sin post-mortem

El error solo se detecta cuando un estudiante lo reporta. No hay monitoreo proactivo.

El problema no es la persona. Es la falta de proceso.

Analizando las causas, vemos cuatro problemas estructurales. Primero, la creación manual —cada condición de acceso se escribe a mano, sin verificar que la actividad destino exista. Segundo, no hay revisión por otro miembro del equipo antes de publicar. Tercero, no hay checklist que garantice que todo está en orden. Y cuarto, no hay monitoreo: el error solo se descubre cuando un estudiante lo reporta. El problema no es la persona que creó el curso. Es la falta de proceso.

El problema inmediato

Diseño vs Técnico: dos mundos, un curso

Hoy hay pasantes de diseño corrigiendo cosas visibles. Los bugs estructurales siguen invisibles. La pregunta es quién arregla qué.

🎨 Pasantes de diseño

Ven y arreglan:

  • Colores, imágenes, textos
  • Formato de actividades
  • Links rotos visibles
  • Consistencia visual entre módulos
✅ Perfecto para diseño.
❌ No pueden ver gates rotos ni deadlocks

🔧 Nosotros (dev)

Detectamos y corregimos:

  • Gates a actividades inexistentes
  • Deadlocks en cascada
  • Recursos sin completion tracking
  • Condiciones de acceso inválidas
  • Dependencias rotas (como curso 269)
✅ validate-deps detecta todo esto en segundos

La solución: un pase de posta simple — cada bug se etiqueta como diseño o técnico.
Los pasantes arreglan diseño. validate-deps encuentra lo técnico. Nosotros corregimos lo que los pasantes no pueden ver.

Hoy la situación es que hay pasantes de diseño trabajando en los cursos. Ellos pueden ver y corregir problemas visuales, pero no pueden detectar bugs estructurales como un gate que referencia una actividad que no existe. La propuesta es simple: establezcamos un pase de posta. Cada bug se etiqueta como diseño o técnico. validate-deps encuentra los técnicos automáticamente.

La solución es simple en concepto: tratemos el curso como código. En lugar de configurar dependencias a mano en Moodle, generamos un blueprint machine-readable —un archivo curso.yaml— que describe toda la estructura del curso. Ese blueprint pasa por validación automática, revisión por pares, y solo entonces se publica. Y después de publicado, lo monitoreamos.

La herramienta que el equipo ya conoce

Gemini CLI: del browser al terminal

El equipo ya usa Gemini en el browser. Gemini CLI es lo mismo, pero con superpoderes: ve archivos, ejecuta comandos, usa MCP, y se configura con un archivo de texto.

🌐 Lo que usan hoy

  • Gemini en el browser
  • Copiar/pegar prompts manualmente
  • Subir archivos uno por uno
  • Sin acceso al file system local
  • Sin herramientas (shell, MCP)

💻 Gemini CLI

  • Mismo modelo Gemini
  • Lee archivos del proyecto directamente
  • Ejecuta comandos en tu terminal
  • Usa MCP servers (GitHub, DBs, APIs)
  • Pegá capturas con Ctrl+V
  • Gratis, open source

Gemini CLI — interfaz oficial (Google)

El equipo ya está familiarizado con Gemini. La diferencia es que hoy usan el browser: copian prompts, pegan resultados, suben archivos manualmente. Gemini CLI es el mismo modelo —Gemini— pero corriendo directo en la terminal. Lee archivos del proyecto, ejecuta comandos, usa MCP servers, y lo más importante: se configura con un settings.json que el unc-course-kit genera automáticamente. El equipo no tiene que aprender nada nuevo —sigue usando Gemini, pero con más herramientas.

El pipeline tiene tres etapas. En pre-build, la validación es automática: el validate-deps corre al generar el blueprint y detecta gates rotos antes de escribir una línea de contenido. En pre-publish, hay controles mixtos: validación automática contra Moodle, más peer review y prueba manual como estudiante. En post-publish, monitoreamos KPIs y recibimos alertas si algo anda mal. La clave está en que cada etapa tiene un gate que impide avanzar si no se cumplió el control anterior.

Curso-as-Código

Peer review como Pull Request

El mismo modelo que GitHub usa para revisar código, aplicado a cursos Moodle.

📝 El prompter escribe

# curso.yaml — sección Módulo 3
unlock_requires:
- item: "Notebook Funcion-Lambda"

👀 El revisor ve el diff

$ git diff curso.yaml
unlock_requires:
- item: "Notebook Funcion-Lambda"
+ item: "Funciones Lambda"
🔍
Revisor corre validate-deps
→
✅
Aprueba o rechaza
→
🚀
Se publica a Moodle
Acá está la innovación clave. En lugar de depender de una sola persona para todo el ciclo, cada cambio al curso pasa por un proceso de revisión como el que usan los desarrolladores para código. El prompter escribe el blueprint. Otro miembro del equipo —el revisor— ve exactamente qué cambió con un git diff, corre la validación automática, y aprueba o rechaza. Esto garantiza que dos pares de ojos vean cada gate, cada condición, cada dependencia, antes de que llegue a producción.

Automatizado

validate-deps detecta el error

El validador corre automáticamente al generar el curso. Este es el output para el curso 269:

🔴 ERROR (1)
"Notebook Funcion-Lambda" no existe en ninguna sección del curso
→ Afecta: Módulo 3 (bloqueado permanentemente)
→ Cascada: Cierre también bloqueado
⚠️ ADVERTENCIA (1)
"Botón al módulo 2" es type=label — no soporta completion tracking
→ Si se usa como gate, nunca se completará
💡 Sugerencia: ¿Querés corregir "Notebook Funcion-Lambda" antes de continuar?
Esto es lo que el validador produce automáticamente. Detecta el gate roto, la cascada, y hasta las advertencias sobre recursos que no soportan completion tracking. Si el equipo hubiera tenido esto antes de publicar el curso 269, el error se habría detectado en el momento cero, antes de que cualquier estudiante lo viera.

Intervención humana

Puntos de control obligatorios

Cada fase del kit tiene un checklist que alguien debe firmar antes de avanzar.

Fase Control Responsable ¿Obligatorio?
01-Fundación Objetivos definidos, narrativa trazada Content lead ✅
02-Scaffold Estructura en Moodle, gates configurados Dev lead ✅
03-Contenido HTML generado, links verificados Prompter ✅
04-Revisión QA + validate-deps + peer review + firma Peer reviewer 🔴 GATE
05-Publicación Subido a Moodle, verificado como estudiante Publisher ✅
06-Mantenimiento KPIs revisados semanalmente Coordinator ✅
🔴 La Fase 04 es el punto de corte: nada se publica sin validate-deps pasado, checklist de QA completo, peer review aprobado y firma del revisor.
El proceso no es solo automático. Definimos puntos de control manual obligatorios en cada fase. El más importante es la Fase 04, Revisión: es el gate que impide publicar si no se cumplen tres condiciones: validate-deps pasado, QA checklist completo, y peer review aprobado con firma. Esto asegura que ningún curso llegue a producción sin al menos dos pares de ojos que lo hayan revisado.
